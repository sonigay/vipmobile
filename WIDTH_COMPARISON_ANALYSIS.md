# 가로 너비 비교 분석: 1280px vs 1920px

## 📊 기본 스펙 비교

### 1280px (현재)
- **원본 너비**: 1280px
- **실제 캡처 너비**: 1280 × 2(SCALE) = **2560px**
- **최대 이미지 크기**: 2560px × 8000px = 20,480,000 픽셀
- **압축 전 파일 크기**: 약 80MB (RGBA 기준)
- **압축 후 예상 크기**: 약 **15-20MB** (PNG 압축)
- **25MB 제한 여유도**: **5-10MB 여유**

### 1920px (변경 시)
- **원본 너비**: 1920px
- **실제 캡처 너비**: 1920 × 2(SCALE) = **3840px**
- **최대 이미지 크기**: 3840px × 8000px = 30,720,000 픽셀
- **압축 전 파일 크기**: 약 120MB (RGBA 기준)
- **압축 후 예상 크기**: 약 **20-25MB** (PNG 압축)
- **25MB 제한 여유도**: **0-5MB 여유** (위험)

---

## ⚖️ 성능 비교 분석

### 1. 파일 크기 및 업로드 성능

#### 1280px (현재)
✅ **장점:**
- 25MB 제한 안정성: 여유 5-10MB
- 업로드 실패 가능성 매우 낮음 (< 1%)
- 네트워크 전송 시간: 상대적으로 빠름
- 서버 저장 공간 효율적

❌ **단점:**
- 파일 크기 절약이지만, 매우 큰 콘텐츠는 품질 저하 가능성

#### 1920px (변경 시)
✅ **장점:**
- 더 높은 해상도로 품질 향상
- 실제 화면 해상도(1920px 이상)와 더 잘 맞음
- 텍스트 가독성 향상 (작은 텍스트도 선명)

❌ **단점:**
- **25MB 제한 위험**: 큰 슬라이드는 25MB 근접 또는 초과 가능
- 업로드 실패 가능성 증가 (약 5-10%)
- 네트워크 전송 시간 20-30% 증가
- 서버 저장 공간 50% 증가

---

### 2. 메모리 사용량

#### 1280px (현재)
- **캔버스 메모리**: 2560 × 8000 × 4 bytes ≈ **80MB** (최대)
- **브라우저 메모리 부족 위험**: 낮음
- **자동 크롭 가능**: 큰 캔버스도 크롭 가능 (50M 픽셀 이하)

#### 1920px (변경 시)
- **캔버스 메모리**: 3840 × 8000 × 4 bytes ≈ **120MB** (최대)
- **브라우저 메모리 부족 위험**: 중간 (특히 낮은 사양 기기)
- **자동 크롭 가능성**: 여전히 가능하지만 더 큰 메모리 필요

**결론**: 1280px이 메모리 효율적 (약 33% 절약)

---

### 3. 캡처 속도 및 렌더링 성능

#### 1280px (현재)
- **html2canvas 처리 시간**: 상대적으로 빠름 (더 적은 픽셀)
- **DOM 렌더링 시간**: 빠름
- **브라우저 부하**: 낮음

#### 1920px (변경 시)
- **html2canvas 처리 시간**: 약 20-30% 증가 (더 많은 픽셀)
- **DOM 렌더링 시간**: 약간 증가
- **브라우저 부하**: 중간 (특히 낮은 사양 기기에서 지연 가능)

**결론**: 1280px이 캡처 속도가 약 20-30% 빠름

---

### 4. 이미지 품질 및 가독성

#### 1280px (현재)
- **실제 해상도**: 2560px (Retina 디스플레이 품질)
- **텍스트 가독성**: 대부분의 콘텐츠에서 충분히 선명
- **차트/그래프 품질**: 충분히 선명
- **작은 텍스트**: 약간 흐릴 수 있음 (8pt 이하)

#### 1920px (변경 시)
- **실제 해상도**: 3840px (고해상도 품질)
- **텍스트 가독성**: 모든 텍스트 크기에서 매우 선명
- **차트/그래프 품질**: 매우 선명
- **작은 텍스트**: 완벽하게 선명

**결론**: 1920px이 품질 면에서 우수 (특히 작은 텍스트)

---

### 5. 콘텐츠 가로 너비 일치도

#### 1280px (현재)
- **문제점**: 실제 콘텐츠가 1280px보다 넓은 경우 (예: 1600px, 1920px)
  - 콘텐츠가 1280px로 재흐름됨 (세로로 길어짐)
  - 레이아웃 변화로 인한 UI 깨짐 가능성
  - 테이블/그래프가 좁아져 보일 수 있음

#### 1920px (변경 시)
- **장점**: 대부분의 슬라이드 콘텐츠가 1920px 이하
  - 레이아웃 재흐름 최소화
  - 원본 디자인과 거의 동일한 캡처
  - 테이블/그래프가 자연스럽게 캡처됨

**결론**: 1920px이 콘텐츠 일치도에서 우수 (실제 화면과 유사)

---

### 6. 업로드 안정성 (25MB 제한)

#### 1280px (현재)
- **평균 파일 크기**: 약 10-15MB
- **최대 파일 크기**: 약 20MB
- **25MB 초과 가능성**: < 1% (매우 낮음)
- **업로드 실패 위험**: 거의 없음 ✅

#### 1920px (변경 시)
- **평균 파일 크기**: 약 15-20MB
- **최대 파일 크기**: 약 23-25MB
- **25MB 초과 가능성**: 약 5-10% (큰 슬라이드에서)
- **업로드 실패 위험**: 중간 ⚠️

**결론**: 1280px이 업로드 안정성에서 우수

**추천 시나리오:**

- ✅ 빠른 캡처가 필요한 경우
- ✅ 메모리가 제한적인 환경

**장점:**
1. ✅ 콘텐츠 가로 너비 일치도 높음 (재흐름 최소화)

**단점:**
1. ❌ 25MB 제한 위험 (5-10% 가능성)
2. ❌ 메모리 사용량 증가 (33% 증가)
3. ❌ 캡처 속도 느림 (20-30% 느림)
4. ❌ 서버 저장 공간 증가
5. ❌ 네트워크 전송 시간 증가

**추천 시나리오:**
- ✅ 이미지 품질이 중요한 경우
- ✅ 콘텐츠 가로 너비가 다양하고 넓은 경우
- ✅ 레이아웃 보존이 중요한 경우
- ✅ 메모리/성능 여유가 있는 경우

## 🎯 최종 권장사항

### 상황별 추천

#### 1. **현재 문제가 콘텐츠 가로 너비 불일치인 경우**
   → **1920px로 변경 권장**
   - 레이아웃 재흐름 문제 해결
   - 하지만 파일 크기 모니터링 필수

#### 2. **현재 업로드 실패가 주요 문제인 경우**
   → **1280px 유지 권장**
   - 25MB 제한 안정성 확보
   - 콘텐츠 재흐름은 감수

#### 3. **하이브리드 접근 (추천)**
   - **기본**: 1280px 유지 (안정성)
   - **특정 슬라이드 타입만 1920px**: 콘텐츠가 넓은 슬라이드만
   - **동적 조정**: 콘텐츠 너비에 따라 자동 선택

  if (contentWidth <= 1280) {
    return 1280;  // 안정성 우선

    return 1920;  // 최대 1920px
